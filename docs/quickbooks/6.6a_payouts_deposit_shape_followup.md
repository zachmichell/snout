# 6.6a — Deposit shape follow-up

## What's now shipped end-to-end

- **Schema**: `processor_payouts`, `payment_payouts`,
  `quickbooks_accounts.default_fee_account_id`,
  `default_bank_account_id`. Plus debug columns
  `last_request_body`, `last_response_body` on `processor_payouts`
  for shape iteration.
- **Stripe ingestion**: `quickbooks-ingest-stripe-payouts` edge function
  pulls payouts from each org's connected Stripe account, builds
  `processor_payouts` rows + `payment_payouts` junctions linked via
  `stripe_payment_intent_id`. Daily cron fires at 03:15 UTC plus a
  manual "Pull from Stripe" button on the QuickBooks settings tab.
- **QBO sync**: `quickbooks-sync-payouts` edge function reads ready
  payouts, builds Deposit bodies (gross from source account + negative
  fee on Fee account), POSTs to QBO.
- **UI**: PayoutsCard with Fee Account picker, "Pull from Stripe", and
  "Sync payouts now" buttons.
- **Cron**: `invoke_quickbooks_payouts_pipeline()` runs ingest + sync
  in sequence.

## Open issue: QBO rejects the Deposit body

Every shape we've tried returns:

> `400 — Business Validation Error: Select a bank account for this
> deposit. (code 6000)`

We've verified:
- DepositToAccountRef value `38` is the Chequing account, AccountType=`Bank`,
  AccountSubType=`Checking`, Active=true, CurrencyRef=CAD,
  SubAccount=false, CurrentBalance=$21,095.57.
- All three accounts in play (Bank, Source = Undeposited Funds, Fee =
  Merchant Account Fees) are CAD, active, not sub-accounts.
- The realm has Multicurrency enabled (HKD/USD parallel A/P + A/R) but
  all our refs are CAD.

The exact payload that gets rejected is captured in
`processor_payouts.last_request_body`. To inspect it:

```sql
select jsonb_pretty(last_request_body), last_response_body
from public.processor_payouts
where state = 'failed'
order by updated_at desc limit 1;
```

## Hypotheses to test in a focused session

1. **QBO Postman against the connected realm**. Open the Intuit
   Developer Playground, select the operator's sandbox, and POST the
   exact captured body to `/v3/company/<realm>/deposit`. If Postman
   accepts it, the issue is in our pg_net path (auth header, token
   refresh). If Postman rejects it, the issue is the body.
2. **Try with `Deposit` API query first** to confirm the entity is
   reachable: `select * from Deposit MAXRESULTS 1`. If the entity
   isn't queryable at all, the realm has restrictive permissions.
3. **Try a no-fee deposit** (single positive line, no negative fee
   line). Validates whether the rejection is about the fee line vs
   the deposit destination.
4. **Try LinkedTxn-style line** instead of AccountRef-style. Pull a
   specific Payment via LinkedTxn rather than summary AccountRef.
   Different code path through QBO's deposit validator.
5. **Check the OAuth scope**. Standard `com.intuit.quickbooks.accounting`
   should cover Deposit, but worth verifying — some Canadian realms
   need explicit additional scopes for cash-handling entities.

## Minimal test invocation

```sql
-- Synthesize a no-fee payout to isolate the fee line
insert into public.processor_payouts
  (organization_id, processor, processor_payout_id, gross_cents,
   fee_cents, net_cents, payout_date, currency)
values (
  'a1b2c3d4-e5f6-7890-abcd-ef1234567890'::uuid,
  'stripe', 'po_minimal_test', 5000, 0, 5000, current_date, 'CAD'
);

-- Trigger the sync (need vault key)
do $$
declare _key text; _req bigint;
begin
  select decrypted_secret into _key from vault.decrypted_secrets where name='service_role_key' limit 1;
  select net.http_post(
    url := 'https://empdnuzfjgfnphwauhah.supabase.co/functions/v1/quickbooks-sync-payouts',
    headers := jsonb_build_object('Authorization', 'Bearer '||_key, 'Content-Type', 'application/json'),
    body := jsonb_build_object('org_id', 'a1b2c3d4-e5f6-7890-abcd-ef1234567890'),
    timeout_milliseconds := 60000
  ) into _req;
end $$;

-- Inspect what got sent and what came back
select state, last_response_body
from public.processor_payouts
where processor_payout_id = 'po_minimal_test';
```

## When this lands

When the Deposit shape works:
1. Remove the `last_request_body` / `last_response_body` columns or
   trim to a smaller diagnostic field.
2. Connect a real Stripe Connect account with actual payouts in
   stripe-mode-test.
3. Verify the daily cron + manual "Pull from Stripe" produce
   correctly-shaped Deposits over the wire.

The integration is otherwise complete — UI, schema, cron, and ingest
all wired. Only the QBO POST body itself needs the last 1% tuning.

# 6.4b — Refund handling via QBO RefundReceipt

Status: planned, not built. Awaiting operator decisions noted below.

## Why this is deferred

Refunds touch accounting policy (which expense account, which deposit
account, partial vs full refund modeling) that varies per operator. The
current worker filters `payments.status IN ('succeeded','refunded')`
already, but the refunded path falls through to the same Payment-create
codepath — which is wrong. Refunds belong on a different QBO entity.

## QBO model

- **Payment** records money received and applies it to one or more
  invoices via `LinkedTxn`. The original invoice flips to "Paid" once
  fully applied.
- **RefundReceipt** records money sent back to the customer. It does
  not link back to the original invoice or payment in QBO; the prior
  Payment stays untouched and the invoice stays "Paid". The
  RefundReceipt itself contributes to the customer's transaction
  history and shows up in P&L as a contra-revenue line.

The two entities are independent. A refunded Snout payment must produce
both: the original Payment so the invoice keeps its paid state, plus a
RefundReceipt for the refund itself.

## Proposed model

`quickbooks_entity_mappings` already supports `(snout_table, snout_id,
qbo_entity_type)` per the existing schema. A refunded payment can have
two rows:

| snout_table | snout_id | qbo_entity_type |
|---|---|---|
| `payments` | `<uuid>` | `Payment` |
| `payments` | `<uuid>` | `RefundReceipt` |

Worker logic for the `payments` branch:

1. Always sync the QBO Payment first (current behavior). Skip if
   already synced.
2. If `payment.status = 'refunded'`, look up the existing Payment
   mapping and confirm it's `synced`. Then check for a RefundReceipt
   mapping. If none, create one.
3. The RefundReceipt's `Line` mirrors the original invoice's lines (so
   the refund can be categorized to the same income accounts and the
   tax breakdown stays correct), with a single sign flip — the line
   amounts are positive (it's a separate transaction), but the
   GlobalTaxCalculation/TaxCodeRef pattern from 6.4.5d still applies.
4. `DepositToAccountRef` on RefundReceipt = same logic as Payment. Use
   the operator's chosen deposit account; if none, omit and let QBO
   default.

## Operator decisions needed

1. **Partial refunds.** If Snout supports partial refunds (e.g.
   `refund_amount_cents < amount_cents`), should the RefundReceipt
   reflect only the refunded amount with a single description-only
   line, or should it itemize prorated portions of each original line?
   Recommendation: single description-only line for partial refunds,
   itemized only for full refunds. Operator confirms.
2. **Same-day reversal vs later refund.** Stripe distinguishes a same-
   day reversal (the original Payment is voided / refunded inline)
   from a later refund (a separate transaction days/weeks later). QBO
   doesn't have an equivalent same-day-reversal concept; it's always
   a separate RefundReceipt. Confirm we don't need to special-case
   this.
3. **Refund deposit account.** Some operators want refunds to debit
   the same account that received the payment (e.g. Undeposited
   Funds), others want a separate "Refunds Issued" bank or clearing
   account. Default: same account. Operator confirms.

## Implementation order

1. Extend `_shared/quickbooks.ts` with `QboRefundReceiptInput`,
   `createRefundReceipt`, `updateRefundReceipt`, `findRefundReceiptByRefNum`.
2. Migration: relax the unique constraint on
   `quickbooks_entity_mappings` to be on `(organization_id, snout_table,
   snout_id, qbo_entity_type)` if not already, so two mapping rows can
   coexist for the same payment.
3. Worker: add the conditional second pass in the `payments` branch.
4. Test: synthesize a refund row in sandbox, run worker, verify
   RefundReceipt exists in QBO with the right amount.

Estimated effort: half a day, gated by operator decisions above.
